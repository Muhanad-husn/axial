# Feature: Structured run logging — one queryable record per run that matters

Every corpus-wide pass today emits to `print()` and gets captured by hand:
`command > data/logs/envelope-regen-235.log 2>&1`. There is **zero `logging`
usage in `src/`** — `cli.py` alone has 49 `print()` calls, `ingest.py` 7,
`tag.py` 5 — and no run leaves a machine-readable record of what it did. The
one exception proves the need: `axial ingest` already appends a per-source TSV
row (source_id, status, notes_count, duration, exit_code, timestamp) to
`data/gold/ingest.results.tsv`. That is a proto-`run.jsonl` bolted onto one
command. This feature generalizes it into a seam every long-running pass shares.

The mandate (issue [#270](https://github.com/Muhanad-husn/axial/issues/270),
founder-set): every run that matters writes into
`data/logs/<name>-<timestamp>/`:

- **`run.jsonl`** — one JSON record per unit of work: `source_id`, `pass`,
  `model`, `status`, `duration_sec`, `error`. Ids and values only.
- **`console.log`** — raw stdout/stderr, unchanged.
- **`summary.md`** — command, counts, outliers, next action. **Operator-authored
  at run end**, not generated by code.

"Runs that matter" = corpus-wide passes, regens, rollouts, evals. **Not** unit
tests, not one-off commands. This is **not a fourth system of record** —
findings and decisions still go in the issue thread (DEC-4). It **adds** a
record; it does **not** change the CLI surface. Console output stays exactly as
it is today.

## The lane decision: slice, not fix-lane

The issue names the prerequisite explicitly: *"decide whether this is fix-lane
or a slice. It touches multiple `src/` modules and adds a seam."* The call is
**slice**, and here is why the `/fix` fast lane (DEC-24) does not fit.

`/fix` is for changes *too small for a slice* — it skips the ceremony (spec,
outer acceptance test, two-stage review) and classifies work as non-behavioral
(fixer alone), a scoped behavioral bug, or feature-scale (bounced to
`/sprint-start`). This work fails the fix-lane test on three counts:

1. **It adds new behavior, not a correction.** There is no bug here. A run dir
   that did not exist gets created; records that were never written get
   appended. New behavior wants a locked outer acceptance test (DEC-1) — the one
   thing `/fix` drops.
2. **It is a new cross-cutting seam with downstream consumers.** `runlog.py` is
   an abstraction every pass calls and that two sibling issues *read*:
   [#277](https://github.com/Muhanad-husn/axial/issues/277) (`axial run`) consumes
   it for corpus passes, and
   [#288](https://github.com/Muhanad-husn/axial/issues/288) reads `run.jsonl` for
   not-applicable/unlisted rates. A seam with named consumers is precisely what a
   fix-lane change is not.
3. **It touches several modules by design** — `extract`, `envelope`, `tag`,
   `eval`. DEC-24 bounces feature-scale, multi-module work back to a sprint.

The phase-a-completion plan already lists this as slice **0b** and calls it *"the
one true serialization point"* — it edits the same passes several other slices
edit, so it lands alone. That is slice discipline, not fix-lane ceremony-dropping.

**But the seam is small.** So it splits into a walking-skeleton slice (the seam
proven through one pass) plus one fan-out follow-on (the remaining three passes).
Two thin slices, not one fat one — which is exactly the "seam is one small slice,
pass-wiring is follow-on" shape the issue anticipates.

## Dependencies

- **01 is the foundation.** It builds the `run_context` contract, the record
  shape, the console-tee, and the clock/run-dir injection seam every later wire
  reuses. 02 cannot start before it.
- **Land this whole feature late in phase-a-completion, alone.** Per the plan's
  Wave 3, 0b (#270) edits the same passes that slices 1a (#278, `envelope`) and
  2a (#294, `tag`) edit. Running it concurrently guarantees conflicts. Land it
  after those passes are stable — ideally just before the stage-4 re-tag, so that
  run is the first fully-logged one.
- **No dependency on siblings, but a shared seam** (see below).

## Boundary with siblings — note, don't build

- **[#277](https://github.com/Muhanad-husn/axial/issues/277) `axial run` (`plans/run/`)**
  is where this logging is *consumed* for corpus passes: the in-process runner
  loop opens one `run_context` and drives every source through it. The **emitter
  lives here**, in `runlog.py`; #277 imports it. This feature does not build the
  runner, and the runner does not build the emitter.
- **[#288](https://github.com/Muhanad-husn/axial/issues/288) not-applicable/unlisted
  rates** *reads* `run.jsonl` records at run end. This feature guarantees the
  record shape and the fields #288 needs (`source_id`, `pass`, `status`, plus the
  per-pass values it counts); it does not compute the rates.

## Out of scope (whole feature)

- **Changing the CLI surface or console output.** Every existing `print()` stays.
  This ADDS `run.jsonl` + a captured `console.log`; it does not restyle, silence,
  or reroute what an operator sees on the terminal today.
- **Generating `summary.md`.** The summary is operator-authored prose (command,
  counts, outliers, next action). The seam creates the run dir so the file has a
  home; it does not write the narrative. A stub header is the ceiling — anything
  more is gold-plating a human's job.
- **A logging framework.** Stdlib `logging` + a `FileHandler` + a thin
  context-manager is the whole mechanism (over-engineering tripwire: no
  structlog, no loguru, no config DSL nobody sets).
- **Wiring every command.** Only the four long-running passes (`extract`,
  `envelope`, `tag`, `eval`), and only after the seam is proven on one. Unit
  tests and one-off commands never open a run dir.
- **A query tool over `run.jsonl`.** #288 reads the records; a general query CLI
  is not in #270.
- **Retention, rotation, or GC of old run dirs.** Orphan cleanup is #291's
  concern, not this seam's.

## Notes / open questions

- **DEC-23 — no source text, ever.** A `run.jsonl` record carries `source_id`,
  `pass`, `model`, `status`, `duration_sec`, and a short `error` string only.
  Never `chunk_text`, never a source passage, never a model raw-response body
  (the #115 post-mortem is why this is load-bearing). The record is ids, values,
  and status. `data/logs/` is under the `data/` gitignore regardless, but the
  no-text rule holds independent of the gitignore.
- **Unit of work = per source, not per chunk.** The four passes run per source;
  `run.jsonl` gets ~30 rows per run, which stays queryable. Per-chunk records
  would be ~17k rows and bloat the file for no operator gain. Per-source is the
  default granularity; a pass may add finer records only with a stated reason.
- **The `model` field is nullable.** `extract` is a structural pass with no LLM
  call, so its records carry `model: null`. That is a feature, not a gap — the
  walking skeleton (01) uses `extract` precisely to prove a model-free record
  round-trips, before 02 adds the model-bearing passes.
- **Reuse the ingest precedent.** `run_ingest` already computes per-source
  `duration_sec` (via `time.monotonic()`) and a status/exit_code row. The record
  shape here is the same fields in JSONL instead of TSV. Where the runner (#277)
  generalizes `run_ingest`, that TSV write folds into `run.jsonl`.
- **Determinism of the run dir.** The dir name embeds a timestamp, so the helper
  takes an injectable clock and an optional explicit run-dir path. Tests pass a
  fixed dir; production passes none and gets `data/logs/<name>-<now>/`. This keeps
  the acceptance test deterministic without freezing wall-clock globally.
